- 软件工程是指应用计算机科学、数学及管理科学等原理，以工程化的原则和方法来解决软件问题的工程
- 目的是提高软件生产率、提高软件质量、降低软件成本
- IEEE定义：将系统的、规范的、可度量的工程化方法应用于软件开发、运行和维护的全过程及上述方法的研究
- 软件工程由方法、工具和过程三个部分组成 
  - 软件工程方法是完成软件工程项目的技术手段，它支持整个软件生命周期；
  - 软件工程使用的工具是人们在开发软件的活动中智力和体力的扩展与延伸，它自动或半自动地支持软件的开发和管理，支持各种软件文档的生成；
  - 软件工程中的过程贯穿软件开发的各个环节，管理人员在软件工程过程中，要对软件开发的质量、进度、成本进行评估、管理和控制，包括人员组织、计划跟踪与控制、成本估算、质量保证和配置管理等。
#### 7.1 软件生命周期
- **软件生命周期过程**（GB/T8566—2022）：
  - 基本过程：获取（需方）、供应（供方）、开发（开发方）、运作（操作方）、维护（维护方）
  - 支持过程：文档编制、配置管理、质量保证、验证、确认、联合评审、审核、问题解决、易用性
  - 组织过程：管理、基础设施、改进、人力资源、资产管理、重用大纲管理、领域工程
- **各阶段任务**：
  1. 可行性研究和项目开发计划：通过分析用户提出的软件开发要求，确定软件项目的性质、目标和规模，得出可行性研究报告。如果可行性研究的结果是可行的，就要制定详细的项目开发计划。这两个活动通常被整合在一起进行，在实际工作中通常把它们归类到同一个阶段中
  2. 需求分析：需求分析工作是软件生命周期中重要的一步，也是决定性的一步。只有通过需求分析，才能把软件功能和性能的总体概念描述为具体的软件需求规格说明，从而奠定软件开发的基础
  3. 概要设计：根据软件需求规格说明建立软件系统的总体结构和模块间的关系，定义各功能模块接口，设计全局数据库或数据结构，规定设计约束，制定组装测试计划
  4. 详细设计：将各模块要实现的功能用相应的设计工具详细描述出来
  5. 实现：写出正确的、易理解的和易维护的程序模块。程序员根据详细设计文档将详细设计转化为程序，完成单元测试
  6. 组装测试（集成测试）：将经过单元测试的模块逐步进行组装和测试
  7. 确认测试：测试系统是否达到了系统需求，按照规格说明书的规定，由用户(或在用户积极参与下)对系统进行验收。必要时，还可以再通过现场测试或并行运行等方法对系统进行进一步的测试
  8. 使用：将软件安装在用户确定的运行环境中，测试通过后移交用户使用。在软件的使用过程中，客户和维护人员必须认真收集发现的软件错误，定期或阶段性地撰写软件问题报告和软件修改报告。
  9. 维护：通过各种必要的维护活动使系统持久地满足用户的需要
  10. 退役：终止对软件产品的支持，软件停止使用

#### 7.2 软件开发方法与模型
- 软件开发方法是指软件开发过程所遵循的办法和步骤
- 软件开发模型则给出了软件开发活动各阶段之间的关系，它是软件开发过程的概括，是软件工程的重要内容。软件开发模型为软件工程管理提供了里程碑和进度表，为软件开发过程提供了原则和方法。
  - 第一种是以软件需求完全确定为前提的瀑布模型;
  - 第二种是在软件开发初始阶段只能提供基本需求时采用的迭代式或渐进式开发模型，例如，喷泉模型、螺旋模型、统一开发过程和敏捷方法等：
  - 第三种是以形式化开发方法为基础的变换模型。
- **传统模型**：
  - **瀑布模型**：线性顺序（软件计划、需求分析、软件设计、 程序编码、软件测试和运行维护），阶段划分，文档驱动
  - **演化模型**：原型迭代，逐步完善
  - **螺旋模型**：风险分析，迭代，四阶段（计划、风险分析、工程、客户评估）
  - **喷泉模型**：面向对象，阶段重叠，无间隙
  - **变换模型**：形式化规格说明，自动变换
  - **智能模型**：基于知识，专家系统
  - **V模型**：测试与开发阶段对应，编码-单元测试（开发者）、详细设计-集成测试、概要设计-系统测试、需求分析-验收测试（业务专家或用户）
- **快速应用开发（RAD，Rapid Application Development）**：
  - 思想：(1)让用户更主动地参与到系统分析、设计和构造活动中来。(2)将项目开发组织成一系列重点突出的研讨会，研讨会要让项目投资方、用户、系统分析师、设计人员和开发人员一起参与。(3)通过一种迭代的构造方法，加速需求分析和设计阶段。（4）让用户提前看到一个可工作的系统。
  - 开发阶段：业务建模、数据建模、过程建模、应用生成、测试与交付
  - 特点：基于构件，快速迭代，适合模块化系统
  - 局限性：不适合高性能、技术风险高的项目
- **统一过程模型（RUP，Rational Unified Process）**：
  - 四个阶段：
    - 初始：明确项目规模；评估项目风险；制订项目计划；阶段技术评审
    - 细化：确定架构；制订构建阶段计划；建立支持环境；选择构件；阶段技术评审
    - 构建：
    - 移交
  - 每个阶段有多个迭代，每个迭代有核心工作流（需求、分析、设计、实现、测试）
  - 特点：用例驱动、以架构为中心、迭代增量
- **敏捷方法**：
  - **敏捷宣言**：个体和交互胜过过程和工具，可工作的软件胜过大量文档，客户合作胜过合同谈判，响应变化胜过遵循计划
  - **主要方法**：
    - **XP（极限编程）**：价值观（沟通、简单、反馈、勇气），实践（结对编程、测试驱动、持续集成、重构、简单设计等）
    - **Scrum**：产品Backlog、Sprint、每日站会、Sprint评审、回顾
    - **DSDM**：用户参与、团队授权、频繁交付、可逆变更、测试贯穿始终
  - **适用场合**：小团队、需求多变、非关键系统

#### 7.3 软件开发环境与工具
- **软件开发环境**：
  - 集成机制：环境信息库、过程控制与消息服务器、环境用户界面
  - ICASE：集成计算机辅助软件工程，信息库方式集成
- **软件开发工具**：
  - 分类：开发工具（需求分析、设计、编码）、维护工具（版本管理、逆向工程）、管理工具（项目管理、配置管理）
  - 选择因素：用户需求、开发人员习惯、项目特点
  - 快速开发工具：可视化编程工具（Visual Studio、Eclipse、IntelliJ IDEA）

#### 7.4 软件过程管理
- **CMM**：
  - 5个成熟度等级：
    1. 初始级：无序
    2. 可重复级：基本项目管理
    3. 已定义级：过程标准化
    4. 可管理级：量化管理
    5. 优化级：持续改进
  - 关键过程域（KPA）列表
- **CMMI**：
  - 阶段式模型：5个等级，过程域分组
  - 连续式模型：过程域按能力等级（0~5）评估
- **软件过程评估**：
  - CMM评估6步骤
  - Trillum模型（嵌入式系统）
  - Bootstrap方法（参考ISO9000-3）
  - ISO/IEC15504标准（过程维+能力维）
  - 我国行业标准SJ/T11234—2001（22个过程域，6个能力等级）

#### 7.5 软件重用和再工程
- **可重用构件**：定义、层次（知识、方法、软件成分）
- **软件重用过程**：
  - 组装模型：查询、修改、组装
  - 类重用模型：实例重用、继承重用、多态重用
  - 软件重用过程模型：领域工程 + 基于构件的软件工程
- **开发可重用构件**：
  - 领域工程三阶段：分析、开发、传播
  - 构件分类：枚举、刻画、属性值
  - 构件重用环境：构件库、管理系统、检索系统、CASE工具
- **逆向工程**：
  - 定义：分析程序，建立高抽象表示
  - 方法：分析信息交换、反汇编、反编译
- **再工程**：
  - 定义：重构既有系统，提高质量
  - 成本：源代码转换 < 结构重构 < 程序和数据重构 < 体系结构迁移

#### 7.6 软件产品线
- **过程模型**：
  - 双生命周期模型：领域工程、应用工程
  - SEI模型：核心资源开发、产品开发、管理（三者互动）
  - 三生命周期模型：企业工程、领域工程、应用工程
- **产品线建立方式**：
  - 演化 vs 革命
  - 基于现有产品 vs 全新产品线
  - 四种方式：将现有产品演化为产品线、用产品线替代现有产品集、全新产品线演化、全新产品线开发

#### 7.7 统一建模语言
- **UML结构**：
  - 构造块：事物（结构、行为、分组、注释）、关系（依赖、关联、泛化、实现）、图
  - 公共机制：规格说明、修饰、公共分类、扩展机制（约束、构造型、标记值）
  - 规则：命名、范围、可见性、完整性、执行
- **系统视图**：逻辑视图、进程视图、实现视图、部署视图、用例视图
- **UML2.0图**（共14种）：
  - 结构图：类图、对象图、构件图、组合结构图、包图、部署图、制品图
  - 行为图：用例图、活动图、状态图
  - 交互图：顺序图、通信图、定时图、交互概览图

#### 7.8 软件形式化方法
- **形式化方法概述**：
  - 基于数学的软件开发方法
  - 优点：精确、可验证
  - 发展趋势：与图形语言结合、CASE工具支持
- **净室软件工程**：
  - 理论基础：函数理论、抽样理论
  - 技术手段：增量开发、基于函数的规范、正确性验证、统计测试
- **程序验证**：
  - 动态方法：测试
  - 静态方法：
    - 静态分析（Coverity）
    - 符号执行（KLEE）
    - 定理证明（Coq、CompCert）
    - 模型检测（SPIN、NuSMV）
